## 📋 各专家意见

### 一、战略与商业层

#### CEO - 周远见
- 观点: Sprint 71 收尾成绩单：P0 完成率 25%（1/4），P1 完成率 12.5%（1/8），P2 完成率 8%（1/13）。这是自 Sprint 65 以来最差的收尾审计。原因不是团队懈怠，而是第132轮 P0 清单过于膨胀——4 个 P0 中 3 个是文档类产出（成本报告、隐私条目、暴力测试），而非代码交付。
- 建议: Sprint 72 收窄 P0 范围至最多 3 项，且必须全部是代码可验证的交付物。文档类产出降级为 P1。
- 优先级: P0

#### COO - 林实干
- 观点: 从第131轮 73% 到第132轮 17%，这下滑曲线比任何 KPI 都刺眼。根因是 Sprint 71 中期加塞了太多 P2 级"锦上添花"项（23→24→25 项连续膨胀），稀释了核心交付。
- 建议: Sprint 72 锁定行动清单上限为 15 项（P0≤3 / P1≤5 / P2≤7）。多出来的推到 Sprint 73。
- 优先级: P0

#### CFO - 钱守正
- 观点: 成本外推报告缺失是最大的财务风险敞口。我们至今不知道全量 302 题插画化的 API 成本。没有这个数字，V9 的商业模式讨论就是空中楼阁。
- 建议: 成本外推报告必须在 Sprint 72 第一周交付，否则 CFO 将行使财务否决权——暂停所有插画化新支出。
- 优先级: P0

### 二、产品与设计层

#### CPO - 叶用户
- 观点: "年龄自适应"从 P2 拖到第133轮还没动工，这是用户体验最大的短板。现在用户每次打开 App 都要手动选年龄——一个 3 岁孩子的妈妈已经选了 30 次"3岁"了，她为什么还要选第 31 次？
- 建议: 年龄自适应（localStorage 记住上次选择 + 首次启动引导）是 P0 级体验债，不能继续拖到 Sprint 73。
- 优先级: P0

#### CTO - 陈架构
- 观点: 测试全绿（67 文件 1921 测试）、lint 零告警——技术债层面 Sprint 71 是健康的。但代码侧交付停滞暴露了流程问题：第132轮 P0 中"成本外推"和"隐私条目"本质上是文档写作任务，却占用了 P0 的工程槽位。工程团队需要的是"可编译、可测试、可部署"的 P0 项。
- 建议: 将"成本外推报告"改为"成本外推计算脚本"（输出 JSON/CSV），"隐私条目"改为"privacy.json 字段新增 + 测试"。让 P0 回归工程交付。
- 优先级: P0

#### UX Director - 苏体验
- 观点: "首页收藏快捷入口"拖了 2 个 Sprint 没落地，全职妈妈小美每次都要点 3 步才能看到收藏。竞争产品（Khan Academy Kids）首页就有"最近收藏"的磁贴。
- 建议: 首页收藏入口用最简单的实现——一个心形图标按钮，点击直接跳 Archive 的 favorites tab。不需要 300ms fade 动画，不需要骨架屏，先让功能可用。
- 优先级: P0

#### 视觉设计总监 - 彩虹姐
- 观点: 年龄色系统一化是视觉一致性的基础债。现在 H5 和 MP 的年龄颜色定义散落在 3 个文件中，任何一个改动都要改 3 处。这在视觉设计上是不可接受的。
- 建议: 在 `content/constants.json` 中新增 `ageColors` 字段，两端统一引用。H5 用 ES import，MP 用 require。这是 30 分钟的工作量，不应该再拖。
- 优先级: P1

- 建议: 少数服从多数——先出插画排期骨架（不含成本），等成本外推报告出来后 48 小时内补上数字。

### 五、合规与风控层

#### 法务张律 - 张律
- 观点: 隐私清单英文版增加"年龄偏好"条目是合规必须项，不是可选项。根据 GDPR-K 和 COPPA，任何存储在设备端且影响儿童内容呈现的偏好设置都必须在隐私政策中声明。虽然 agePreference 只存在 localStorage 且不上传，但"存在"本身就是需要披露的事实。
- 建议: 在 privacy.json 和 privacy-policy.md 中同步增加"Age Preference (localStorage only)"条目。中文版同步更新。
- 优先级: P0

#### 内容安全审核官 - 安全李姐
- 观点: 情绪安全巡检脚本的语境分析已落地（V8.68），本轮巡检发现：黑名单命中 6 题（含 staging），人工审核 4 题（含"疼"），深度审查 1 题（body-005 "为什么流血？"）。其中"为什么不能打人？"（society-014）和"医生为什么要听我的肚子？"（society-018）是黑名单误报——"打人"是行为教育、不是恐怖内容，"医生"是中性的职业名词。建议在巡检脚本中增加"教育语境白名单"。
- 建议: 巡检脚本增加教育语境白名单（如"为什么不能打人"→教育意图放行，"医生"在非医疗恐惧语境下放行）。body-005 "为什么流血？"需要重新评估 Layer 1 回答中的"流血"表述是否需要改为"破皮"。
- 优先级: P1

#### 社会学刘教授
- 观点: 角色多样性审计初步数据：302 题中 62% 提到"妈妈"，18% 提到"爸爸"，仅 8% 提到奶奶/外婆/爷爷/外公，12% 无特定角色。这个比例严重失衡。中国有 30% 的儿童由祖父母抚养，但我们的内容几乎不反映他们的日常经验。
- 建议: 新内容入库模板增加"角色多样性"字段（≥40% 非"妈妈"），Sprint 72 完成 10 题已有内容的角色改写。优先级从 P2 升级为 P1。

### 六、用户反馈与质量控制层

#### 毒舌老王
- 观点: 恭喜团队，Sprint 71 创造了新的"最差收尾率"记录——17%。但我不骂人，因为问题不在团队，在流程。第132轮 P0 清单里塞了 3 个文档任务，这就像让厨师去写菜谱而不是做菜。修复方案很简单：P0 = 可运行代码 + 可执行测试 + 可部署配置。其他的都是 P1。
- 建议: 建立 P0 定义门禁：任何 P0 项必须满足"可编译/可测试/可部署"三个条件之一。不满足的自动降级为 P1。
- 优先级: P0

#### 测试虫虫
- 观点: 67 文件 1921 测试全绿，但有一个隐藏问题：`prd_update_gate.py` 在测试中会打印 `[prd_update_gate] Running npm run lint...` 三次，这表示在 vitest 的 environment 初始化阶段，门禁脚本被重复调用了。不是 bug，但暴露了测试隔离性不够——如果未来门禁脚本有副作用（如写文件），可能导致测试污染。
- 建议: 在 vitest 配置中增加 `globalSetup` 钩子，确保门禁脚本只在测试套件启动时运行一次，而非每个 worker 都跑。
- 优先级: P2

#### 全职妈妈小美
- 观点: 我最需要的是"上次看到哪了"——有时候孩子问了一半被叫去吃饭，回来就找不到刚才的问题了。首页收藏入口是好的，但更好的体验是"继续上次的探索"。
- 建议: 首页增加"继续探索"模块——显示最近浏览的 3 个问题（localStorage，不跨设备）。这比收藏入口更直接地解决"中断-恢复"场景。
- 优先级: P1

#### Global 何
- 观点: 英文翻译目前只覆盖了 30 题，但"年龄偏好"隐私条目和 V9 完整隐私政策英文版是出海合规的基础。建议先出隐私条目（1 小时工作量），再逐步推进完整隐私政策翻译。
- 建议: 优先完成隐私清单英文版"年龄偏好"条目（中英文同步），V9 完整隐私政策英文版排到 Sprint 73。
- 优先级: P0
